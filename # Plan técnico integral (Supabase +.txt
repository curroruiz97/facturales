# Plan técnico integral (Supabase + Resend): envío de email al emitir factura/presupuesto

## Objetivo
Implementar mailing transaccional usando **Supabase Edge Functions + Resend** para que, al emitir una factura o presupuesto, se envíe automáticamente un email al cliente con:
- asunto y cuerpo breve,
- PDF adjunto,
- trazabilidad completa (estado, errores, reintentos).

> Alcance: plan de implementación por fases, minimizando riesgo de regresión.

---

## 1) Estado actual del proyecto (diagnóstico)

1. El flujo de emisión de factura desde `invoices/preview.html` solo llama `emitInvoice(...)` y redirige a `issued.html`.
2. `emitInvoice` y `emitQuote` únicamente actualizan `status = 'issued'` en base de datos.
3. El PDF ya se genera en frontend y existe `getPDFBlob()` para reutilizar en el envío.
4. En Supabase local hay `inbucket` para pruebas y SMTP de auth comentado (no es mailing transaccional de facturas).
5. Hay inconsistencia en presupuesto (`preview.html?quote=...` vs parser actual de preview).

---

## 2) Arquitectura objetivo (cerrada a Supabase + Resend)

## 2.1 Componentes
- **Frontend** (emisión factura/presupuesto): dispara emisión y luego envío.
- **Supabase Edge Function**: `send-document-email`.
- **Resend API**: proveedor único de correo transaccional.
- **Supabase Postgres**: tabla `document_email_log` para auditoría.
- **Supabase Storage (privado)**: opcional en fase avanzada para PDF canónico.

## 2.2 Flujo objetivo
1. Usuario pulsa **Emitir**.
2. Frontend emite documento (`status=issued`).
3. Frontend invoca `supabase.functions.invoke('send-document-email', payload)`.
4. Edge Function valida JWT, ownership, estado `issued` y envía por Resend.
5. Edge Function registra resultado en `document_email_log`.
6. UI muestra: enviado / pendiente / error con opción de reintento.

## 2.3 Principios
- Secretos (`RESEND_API_KEY`) solo en Edge Function.
- Emisión no se revierte si falla email.
- Idempotencia para evitar duplicados.
- Auditoría completa de cada intento.
