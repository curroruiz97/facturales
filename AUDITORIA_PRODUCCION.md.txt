# Auditoría de preparación para producción

## Hallazgos críticos (bloqueantes)

1. **Colisión global de numeración de facturas/presupuestos en entornos multi-tenant**
   - `invoice_number` y `quote_number` están definidos como `UNIQUE` global (tabla completa), pero su generación se calcula por `user_id` + serie + año.
   - Resultado: dos usuarios diferentes pueden generar el mismo número lógico (por ejemplo `A-2026-00001`) y el segundo insert fallará por restricción única global.
   - Impacto: errores de creación en producción, incidencias difíciles de reproducir, pérdida de confianza en el flujo core de facturación.
   - Recomendación: reemplazar la unicidad global por índice compuesto (`user_id`, `invoice_number`) y (`user_id`, `quote_number`) o definir una secuencia robusta por tenant/serie con bloqueo explícito.

2. **Riesgo de XSS almacenado en la gestión de clientes (`users.html`)**
   - Se renderizan campos controlados por usuario (`nombre_razon_social`, `identificador`, `email`, `telefono`, etc.) mediante `innerHTML` sin sanitización.
   - También se interpolan valores en atributos inline (`onclick`) y el escape de comillas es parcial.
   - Impacto: ejecución de JavaScript arbitrario, robo de sesión/tokens, escalado de privilegios en front y compromiso de datos.
   - Recomendación: migrar a renderizado seguro con `textContent` + `addEventListener`, eliminar handlers inline, sanitizar server/client (DOMPurify u otra estrategia equivalente).

3. **Build de producción roto (pipeline de release no confiable)**
   - El build falla por una entrada inexistente (`transaction.html`) declarada en `vite.config.js`.
   - Además, hay un error de parseo HTML por caracteres corruptos en `statistics.html`.
   - Impacto: no hay artefacto reproducible de producción; despliegues fallidos o manuales, mayor riesgo operativo.
   - Recomendación: limpiar `rollupOptions.input`, corregir encoding UTF-8 en HTML y agregar check obligatorio de CI (`npm run build`) antes de merge/deploy.

## Hallazgos de riesgo alto (no bloqueantes inmediatos, pero importantes)

4. **Uso de credenciales Supabase embebidas en código cliente**
   - URL y `anon key` están hardcodeadas en `assets/js/supabaseClient.js`.
   - Aunque la `anon key` no es secreto absoluto en Supabase, mantenerla hardcodeada dificulta rotación y separación de entornos.
   - Recomendación: migrar a variables de entorno de Vite (`import.meta.env`) y estrategia de configuración por ambiente.

5. **Restricciones de unicidad potencialmente globales para datos de negocio multi-tenant**
   - Ejemplo: `clientes.identificador` y `business_info.nif_cif` en migraciones están como `UNIQUE` global.
   - Si el modelo SaaS permite datos repetidos entre tenants, esto provocará falsos conflictos entre empresas distintas.
   - Recomendación: revisar reglas de negocio y, si aplica, pasar a índices compuestos con `user_id`.

## Cosas buenas (fortalezas)

1. **Modelo de seguridad base bien encaminado con RLS por usuario**
   - Varias tablas de dominio aplican `ENABLE ROW LEVEL SECURITY` y políticas con `auth.uid() = user_id`.
   - Esto es una base sólida para aislamiento de datos por tenant.

2. **Uso consistente de `updated_at` con triggers y checks de integridad de dominio**
   - Hay triggers para sellado temporal y validaciones de estado (ej. facturas/presupuestos emitidos no editables en campos críticos).
   - Este enfoque reduce corrupción de datos y ayuda a trazabilidad.

3. **Cobertura funcional amplia del dominio de facturación**
   - Se observan módulos claros para auth, clientes, facturas, presupuestos, onboarding y transacciones.
   - La estructura sugiere buena separación de responsabilidades para iterar producto.

## Priorización recomendada (orden de ejecución)

1. **Semana 1 (inmediato)**
   - Corregir build roto (`vite.config.js`, encoding HTML) y bloquear deploy sin build verde.
   - Mitigar XSS en `users-page.js` (y auditar otros módulos con `innerHTML`).

2. **Semana 1-2**
   - Corregir esquema de numeración/índices únicos de facturas y presupuestos.
   - Ejecutar migración de datos y pruebas de concurrencia (insert simultáneo).

3. **Semana 2**
   - Revisar todas las constraints `UNIQUE` multi-tenant.
   - Externalizar configuración Supabase por entorno.

4. **Semana 3**
   - Hardening adicional: CSP, auditoría de dependencias y pipeline CI/CD con checks automáticos (lint, build, smoke tests).
